# Heal PR Review Workflow

> **Status**: documentation for heal-broaden v1
> **Related**: docs/design/heal-broaden.md §9 OQ#5
> **Audience**: developers, maintainers, operators reviewing heal-generated PRs

## Overview

The heal agent (`agents/heal/POST.md`) spawns Claude Code sessions to address structural gaps in the system — parser misses, missing detector fields, capability seed drifts, and other improvements. When heal proposes a fix, it opens a draft PR against the upstream repository for human review and merge.

This document defines the SLA, review process, and merge criteria for heal-generated PRs.

## PR Lifecycle

### 1. Draft Opening

**Timing**: When heal's Claude Code session successfully builds and tests a fix.

**Process**:
- heal calls `spawn-claude-code-fix` tool
- Tool verifies success criterion (tests pass, build succeeds, etc.)
- Tool invokes `gh pr create --draft` with:
  - Branch: `work/heal-<finding_id>`
  - Title: `heal: <finding_class> for <app_name> [#<finding_id>]`
  - Body: includes finding text, success criterion verified, and transcript path
  - Status: **DRAFT** (not ready for review)

**Example PR title**: `heal: log_format_drift for auth-service [#finding-corpus-001]`

### 2. Reviewer Assignment

| Repository Class | Owner | SLA | Notes |
|---|---|---|---|
| **mallcop** (OSS CLI) | baron | 24h | Project-owned; repo maintainer reviews. |
| **connector-*** (OSS connectors) | baron | 48h | Project-owned; connector maintainer reviews. |
| **mallcop-legion** (agent system) | baron | 24h | Project-owned; internal infrastructure. |
| **External upstream** (GitHub API client, etc.) | TBD | No SLA | If heal is ever configured to target external repos, upstream maintainer reviews. No guaranteed merge window. |

### 3. Promotion to Ready

**Who**: The assigned reviewer.

**Trigger**: Reviewer has reviewed the diff and is satisfied with scope, quality, and testing.

**Action**:
```bash
gh pr ready <pr-number>
```

This promotes the PR from DRAFT to READY_FOR_REVIEW. The GitHub UI surface changes; checks can now run if configured.

### 4. Merge Criteria

**Before merging, verify**:

1. **Scope matches finding**: The diff addresses only the structural gap described in the finding, no scope creep.
2. **Tests pass**: All CI checks green (build, unit tests, integration tests per repo).
3. **No merge conflicts**: Branch is up-to-date with origin/main or rebase is clean.
4. **Author reviewed**: The reviewer has read the code diff and signed off.

**Forbidden**:
- Auto-merge (only human-initiated merge allowed).
- Squash-and-merge if commit history is meaningful (preserve commits if they tell a story; default to squash for single-commit fixes).
- Force-push (should not be necessary; if it is, re-review the branch).

**Merge command**:
```bash
# Default: squash commit history into one.
gh pr merge <pr-number> --squash

# Preserve commits if meaningful history:
gh pr merge <pr-number> --rebase
```

### 5. Failure Handling

**If PR sits beyond SLA without review**:

1. The heal operator monitoring infrastructure (not built yet; future) detects the stale PR.
2. The operator is messaged via `message-operator` (existing F1G tool) with category `open-question`.
3. The next heal-generated PR for the same finding class **waits** until the previous PR is closed (success or failure), to avoid PR flood.
4. The operator decides: merge, close, or request heal to re-do the work with different approach.

**If healing fails** (Claude Code exits non-zero, timeout, etc.):

1. heal closes the finding with verdict `escalated` and reason `attempt_failure_reason=<...>`.
2. Transcript is saved to `.run/transcripts/<run_id>/heal-<finding_id>.jsonl` for operator review.
3. The finding waits for operator intervention; no automatic retry.

## User-Visible PR Body Template

When heal opens a draft PR, the body includes:

```markdown
## Finding

**ID**: <finding_id>
**Class**: <finding_class>
**Description**: <finding description from reader>

## Change

This PR fixes the structural gap by:
- <change 1>
- <change 2>
- ...

## Success Criterion Verified

```
<success criterion text (e.g., "pytest tests/test_parser.py")>
```

Result: PASSED

## Healing Session

Transcript: `/path/to/heal-<finding_id>.jsonl`
Model tier: sonnet
Tokens used: 45,234
Wall time: 312 seconds

## Approvers

@baron — repository owner

---

*This PR was generated by heal (mallcop-legion agents/heal/POST.md). Approval and merge authority rest with the repository maintainer.*
```

## Example Workflows

### Workflow 1: Happy Path (mallcop repo)

```
1. Finding: log_format_drift in auth-service
2. heal spawns Claude Code
3. Session adds log pattern to parser.yaml
4. Tests pass
5. PR opened: "heal: log_format_drift for auth-service [#finding-corpus-001]"
   Status: DRAFT
6. Reviewer (baron) reads PR within 24h
7. Reviewer promotes to READY
8. Checks run (tests, build pass)
9. Reviewer merges with `gh pr merge --squash`
10. PR closed, finding closed as `resolved`, PR URL recorded in finding annotation
```

### Workflow 2: Scope Creep Detected (mallcop repo)

```
1. PR opened by heal with 3 files changed
2. Reviewer notices file outside parser.yaml scope (e.g., detector.py)
3. Reviewer posts review comment: "Out of scope for this finding"
4. Reviewer closes PR with "not-as-intended" label
5. Operator investigates via transcript
6. Finding remains escalated, awaits operator decision on next heal attempt
```

### Workflow 3: SLA Miss (mallcop repo)

```
1. PR opened on 2026-04-29T10:00Z
2. 24h passes (2026-04-30T10:00Z)
3. PR still draft, not promoted
4. Operator alert (future): "Heal PR stale for 24h: <pr-number>"
5. Operator promotes PR, reviews, merges OR decides to close
6. If closed: operator may request heal to retry with different approach
```

## Repo-Specific Notes

### mallcop (OSS CLI)

- Maintainer: baron
- Review SLA: 24h
- Typical changes: parsers (parser.yaml), detectors (detector classes), skills, connectors
- Merge strategy: squash (one logical fix per PR)
- Automation: CI runs on ready PRs only

### connector-* (OSS connectors)

- Maintainer: baron (or delegated connector owner)
- Review SLA: 48h (longer because connector repos may have lower velocity)
- Typical changes: schema extensions, field mappings, auth logic
- Merge strategy: squash unless commit history is pedagogical
- Automation: CI runs on ready PRs only

### mallcop-legion (agent system)

- Maintainer: baron
- Review SLA: 24h
- Scope: `agents/*/POST.md`, `chart/*`, `prompts/*` only (enforced by tool)
- Typical changes: agent behavior tweaks, chart config, prompt refinements
- Merge strategy: preserve commits if meaningful progression
- Automation: CI runs on ready PRs only

## Future Enhancements

### v2 — Stale PR Detection

Implement background task that polls open heal-generated PRs and alerts if SLA exceeded.

### v2 — Automatic Promotion Rules

For low-risk changes (e.g., parser additions that don't touch detector logic), allow heal to promote draft → ready once tests pass.

### v3 — Cross-Repo PR Aggregation

If heal targets multiple repos in parallel, aggregate PR status in a single dashboard for operator visibility.

### v3 — Retry with Escalation

If a healing attempt fails, allow operator to trigger a retry with different model tier (e.g., opus instead of sonnet) or additional context from prior transcript.
